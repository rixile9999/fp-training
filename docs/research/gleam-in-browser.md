# Research: gleam-in-browser

Running Gleam fully in-browser is a solved, officially-supported pattern, verified June 2026. The Gleam compiler (Rust) is published as a WASM artifact attached to every GitHub release of gleam-lang/gleam as `gleam-vX.Y.Z-browser.tar.gz` (latest: v1.17.0, released 2026-06-02, asset is 1.66 MB). It is built with `wasm-pack build --release --target web` from the `compiler-wasm` crate; the tarball is the wasm-pack `pkg/` output (`gleam_wasm.js` JS glue + `gleam_wasm_bg.wasm`). There is no official npm package — you download the release tarball (the tour's `bin/download-compiler` script is a 10-line curl|tar).

The WASM JS API (compiler-wasm/src/lib.rs) is a project-scoped virtual filesystem: `initialise_panic_hook(debug)`, `write_module(project_id, module_name, code)` (writes to /src), `write_file`/`write_file_bytes`/`read_file_bytes`, `compile_package(project_id, target) -> Result<(), String>` (target "javascript" or "erlang"), `read_compiled_javascript(project_id, module_name) -> Option<String>`, `read_compiled_erlang`, `reset_filesystem`, `delete_project`, `reset_warnings`, `pop_warning -> Option<String>`.

tour.gleam.run (gleam-lang/language-tour) and play.gleam.run (gleam-lang/playground, pinned v1.15.2) are static sites (GitHub Pages, generated by a Gleam script). Architecture: a Web Worker (`static/worker.js`) imports `compiler.js`, which dynamically imports `/compiler/gleam_wasm.js` and wraps the WASM API in Compiler/Project classes. Stdlib handling is two-sided: (1) at build time, `src/tour.gleam` reads gleam_stdlib's `.gleam` sources from `build/packages/gleam_stdlib/src/gleam` and embeds them into a generated `stdlib.js` (module-name -> source map); the worker writes every stdlib module into the WASM project via `write_module`, so stdlib is type-checked/compiled as part of the user's package; (2) the already-compiled stdlib `.mjs` files from `build/dev/javascript/gleam_stdlib` are copied to `public/precompiled/`, and after compiling user code the worker regex-rewrites relative imports (`from "./..."` -> `from "<base>/precompiled/..."`). Execution: compiled JS is an ES module (can't eval), so it's base64-encoded and `await import("data:text/javascript;base64,...")`, then `main()` is called. Output is captured by monkey-patching `console.log` inside the worker; runtime exceptions are caught and `toString()`ed; warnings drained via `pop_warning`. The main thread debounces (200ms), keeps one queued job, and has no timeout/kill for infinite loops.

Dependencies: the WASM API has no package/dependency concept (open issue gleam-lang/gleam#3245, "help wanted"). Workaround proven by LiveCodes: any pure-Gleam hex package (gleam_json, gleam_http, gleam_otp, etc.) can be included by writing its sources as project modules plus serving its precompiled JS/FFI files — same pattern as stdlib.

Diagnostics: `compile_package` errors and `pop_warning` return pretty-printed plain-text strings (`Error::pretty_string()` uses `Buffer::no_color()` — no ANSI). They include file:line:column, source excerpt with caret underlines, explanation, and hints — excellent for display, but not structured. `gleam_core::Error` has `to_diagnostics() -> Vec<Diagnostic>` with spans/labels/hints, but compiler-wasm does not expose it; a small fork adding a JSON-serializing export would give structured spans.

Server-side (Docker + gleam CLI) alternative: gives real hex dependency resolution, `gleam test`, and Erlang target execution, but requires sandboxing untrusted code, adds latency/cost/ops for a solo dev. The official tour deliberately compiles "entirely within the browser, rather than sending the code to a build server"; in-browser is free to host as a static site and is the proven path.

## Key facts

- The Gleam compiler is officially distributed as a browser WASM build attached to every GitHub release: asset name gleam-vX.Y.Z-browser.tar.gz; latest is v1.17.0 (released 2026-06-02), asset size 1.66 MB. No official npm package exists; distribution is GitHub release tarballs only.
- The tarball is wasm-pack output (compiler-wasm crate, built with `wasm-pack build --release --target web`, then `tar -C pkg/ -czvf gleam-vX-browser.tar.gz`), containing gleam_wasm.js glue and gleam_wasm_bg.wasm; you load it with `const wasm = await import('.../gleam_wasm.js'); await wasm.default();`.
- WASM JS API (compiler-wasm/src/lib.rs, project-id-scoped in-memory filesystem): initialise_panic_hook(debug: bool); reset_filesystem(project_id); delete_project(project_id); write_module(project_id, module_name, code) writes to /src; write_file / write_file_bytes / read_file_bytes; compile_package(project_id, target: 'javascript'|'erlang') -> Result<(), String>; read_compiled_javascript(project_id, module_name) -> Option<String>; read_compiled_erlang; reset_warnings; pop_warning(project_id) -> Option<String>.
- tour.gleam.run and the playground are fully static sites on GitHub Pages — zero backend. The site generator is itself a Gleam program (`gleam run` builds public/), and `bin/download-compiler` fetches the pinned compiler tarball into wasm-compiler/ which is copied to public/compiler/. language-tour main pins compiler v1.15.2; playground repo GLEAM_VERSION is v1.15.2.
- Stdlib in-browser mechanism (language-tour): build step embeds all gleam_stdlib .gleam sources (from build/packages/gleam_stdlib/src/gleam, skipping deprecated bit_string/bit_builder/map modules) into a generated stdlib.js source map; the Web Worker writes each one via write_module so user code type-checks against real stdlib. Separately, precompiled stdlib .mjs files (from build/dev/javascript/gleam_stdlib) plus FFI .mjs files are copied to public/precompiled/ for runtime imports.
- After compilation the worker rewrites imports in the emitted JS with `editedJs.replaceAll(/from\s+"\.\/(.+)"/g, `from "${href}precompiled/$1"`)` so stdlib imports resolve to the precompiled files served statically.
- Execution: Gleam emits ES modules which cannot be eval()'d, so the tour base64-encodes the JS and runs `await import('data:text/javascript;base64,' + encoded)` inside the Web Worker, then calls the module's exported main(). Output capture = monkey-patched console.log accumulating into a string (covers io.println/io.debug on the JS target); runtime exceptions caught via try/catch and stringified; result {log, error, warnings} posted back via postMessage.
- Sandboxing is just a same-origin Web Worker (not an iframe); the main thread debounces edits at 200ms and keeps only the latest queued job, but has NO timeout/terminate logic — an infinite loop permanently hangs the tour's worker. A real platform should add worker.terminate() + respawn on a watchdog timer.
- Third-party Hex packages: the WASM API has no dependency mechanism (open issue gleam-lang/gleam#3245, labeled 'help wanted'). Proven workaround (used by LiveCodes, which ships gleam_json, gleam_http, gleam_fetch, gleam_otp, gleam_javascript, gleam_erlang, gleam_crypto): write the package's .gleam sources into the same project via write_module/write_file for type-checking, and serve its precompiled .mjs (including any FFI files) for runtime — i.e., treat dependencies exactly like the stdlib is treated.
- Diagnostics: compile_package errors and warnings come back as plain-text pretty-printed strings (Error::pretty_string() uses termcolor Buffer::no_color(), so no ANSI codes). The text includes file path, line:column, a source excerpt with caret underlines, error title/explanation, and optional hints. Structured diagnostics exist in gleam_core (Error::to_diagnostics() -> Vec<Diagnostic> with spans, labels, hints, extra_labels) but are NOT exposed over the WASM boundary — getting machine-readable spans requires a small fork of compiler-wasm (the crate is tiny: one lib.rs plus wasm-pack config) or regex-parsing the pretty text (the tour itself highlights 'src/main.gleam:\d+' patterns with regex).
- Erlang target: read_compiled_erlang exists, but the browser cannot execute Erlang — in-browser execution is JavaScript-target only.
- Compilation runs off the main thread (Web Worker) by design to keep the UI responsive; stdlib (~50 modules) is recompiled inside the WASM project together with user code each session, which the tour does without noticeable issue.

## Recommendations

- Copy the language-tour architecture wholesale — it is small, MIT-style open source, and battle-tested: download gleam-vX-browser.tar.gz at build time, serve it statically, run compile+execute in a Web Worker, execute via base64 data-URL dynamic import, capture console.log. Read static/compiler.js (~90 lines) and static/worker.js (~80 lines) in github.com/gleam-lang/language-tour as the reference implementation.
- Pin a specific compiler version (like the tour pins v1.15.2 against latest v1.17.0) and the matching gleam_stdlib version; bundle stdlib sources + precompiled .mjs at build time from a local `gleam build` of a dummy project, exactly as src/tour.gleam's generate_stdlib_bundle/copy_compiled_stdlib do.
- Add what the tour lacks and a training platform needs: (a) a watchdog that worker.terminate()s and respawns on timeout to survive infinite loops in user drills; (b) structured error spans — fork compiler-wasm (single lib.rs) to add an export like `read_diagnostics(project_id) -> JsValue` serializing gleam_core's Vec<Diagnostic> (title, text, hint, path, SrcSpan start/end) to JSON, enabling inline squiggles and pedagogical error explanations; alternatively parse the stable pretty-text format ('error: ...', 'src/main.gleam:LINE:COL', hint lines) as a no-fork stopgap.
- For exercises needing more than stdlib (e.g. gleam_json for a lesson), use the LiveCodes pattern: vendor the package's .gleam sources into the WASM project as extra modules and serve its precompiled JS/FFI; do not wait for upstream dependency support (issue #3245 is still open as of June 2026).
- Skip server-side compilation for v1: the official tour/playground prove browser-only works, it costs nothing to host (static site / GitHub Pages), has zero per-user latency or abuse surface, and suits a solo developer. Reconsider a Docker sandbox only if you later need the Erlang target, real `gleam test` runs, arbitrary Hex deps, or hidden test harnesses that must not ship to the client.
- Run grading/test assertions in-browser by compiling a hidden test module alongside the user's module in the same WASM project (write_module supports multiple modules), calling its main() and inspecting captured output or thrown exceptions.

## Risks / limitations

- No structured diagnostics over the WASM API today — only pretty strings; the fork-to-expose-spans recommendation adds a Rust/wasm-pack build step to maintain across compiler upgrades.
- No dependency/package support in compiler-wasm (issue #3245 open); the vendor-sources workaround compiles dependency code as part of the user package, so dependency warnings can leak into user-visible warnings and version mismatches are on you.
- Web Worker is a responsiveness boundary, not a security sandbox: compiled user code runs same-origin with network access (fetch) available; acceptable for self-authored code, but consider a sandboxed iframe + worker if you ever execute shared/other-users' code.
- Infinite loops/heavy recursion hang the worker (tour has no watchdog) — must implement terminate/respawn, which also re-pays worker startup (WASM init + stdlib write_module) each kill.
- JavaScript target only in-browser; semantics that differ on the Erlang target (e.g. integer size, concurrency/OTP lessons) cannot be demonstrated client-side.
- The ~1.7 MB WASM download plus stdlib bundle is a real first-load cost on slow connections; cache aggressively (the tour uses a ?v= cache-buster query).
- Minor unverified details: exact in-browser compile latency was not measured, and the precise file list inside the v1.17.0 tarball was inferred from the compiler-wasm README's wasm-pack workflow rather than extracted.
